# pasgobject — Architecture

## Overview

pasgobject is structured as a monorepo with three independent layers:

```
gobject/   — GLib/GObject core binding (hand-written, stable)
gtk4/      — GTK4 widget bindings (auto-generated from Gtk-4.0.gir)
tools/     — Code generation tooling (gir2pas)
```

The `gobject/` layer is the foundation. The `gtk4/` layer depends on it. Nothing
in `gobject/` depends on GTK.

## Dependency graph

```
PasGObject (umbrella)
  └── GSignal
        └── GObject
              ├── GValue
              ├── GParam
              └── GObjectType
                    └── GLib
                          └── GTypes
                                └── GLibFFI (Internal)
                                └── GObjectFFI (Internal)

PasGTK (umbrella)
  └── PasGObject (all of the above)
  └── GtkClasses (generated)
        └── GtkTypes  (generated)
        └── GtkFFI    (generated, Internal)
```

## gobject/ layer

The GObject binding is hand-written Pascal. It implements:

- **Ownership semantics** per the GObject Introspection specification:
  - `TGObject.Take(ptr)` — transfer:full (takes an existing reference)
  - `TGObject.Borrow(ptr)` — transfer:none (adds toggle ref, caller keeps ref)
  - `TGObject.Create` — allocates a new instance via `g_object_new`
- **Toggle-ref lifecycle** — the Pascal wrapper holds one toggle ref. When the
  GObject's total ref count drops to one (the toggle ref), the Pascal object is
  the last owner and will be freed when `Free` is called.
- **Floating reference handling** — objects inheriting `GInitiallyUnowned` (all
  GTK widgets) start with a floating ref. Both `Create` and `CreateFromHandle`
  check `g_object_is_floating` at runtime and call `g_object_ref_sink`
  automatically, so callers never need to know whether a type is floating.
- **Lazy GType registration** — Pascal subclasses of `TGObject` register their
  GType on the first call to `TypeID`. No explicit registration call is needed.
- **Property system** — `DoGetProp`/`DoSetProp` virtual methods. The
  `TGValue.InitFromRaw` stores a pointer to the GLib-provided `GValue` (not a
  copy), so writes inside `DoGetProp` go to the struct GLib owns.
- **Signals** — `SignalConnectMethod` connects an `of object` Pascal method to
  a GObject signal. The closure stores the method pointer in a heap-allocated
  `TGSignalMethodData` record that is freed by `GClosureNotify` on disconnect.

## gtk4/ layer

The GTK4 binding is generated by `tools/gir2pas/gir2pas` from
`/usr/share/gir-1.0/Gtk-4.0.gir`. Three files are produced:

| File | Content |
|---|---|
| `Internal/GtkFFI.pas` | Raw `external` declarations for every GTK4 C function |
| `GtkTypes.pas` | Enumerations and bitfields (110 types) |
| `GtkClasses.pas` | Pascal wrapper classes for all 265 GTK4 classes |

Generated classes:
- Inherit from `TGObject` (or the correct GTK parent class)
- Override `TypeID` to return the GTK GType via the generated `get_type` call
- Expose constructors that call `CreateFromHandle` with the C constructor result
- Expose methods that forward to the FFI layer, converting types as needed

## tools/ layer

`gir2pas` is a standalone FPC program. It:

1. Parses a GIR XML file with FPC's built-in `DOM`/`XMLRead`
2. Extracts namespace, shared library, enumerations, and classes
3. Topologically sorts classes so parent classes always appear before children
4. Emits three Pascal units per namespace

`gir2pas` can generate bindings for any GIR namespace:

```bash
./build/tools/gir2pas/gir2pas /usr/share/gir-1.0/Adw-1.gir adw/src
```

## FPU exception mask

GTK4's rendering stack (Cairo, Pango, GL) performs IEEE 754 operations that
generate FPU exceptions under Free Pascal's default exception mask. Every
program using GTK4 must call:

```pascal
SetExceptionMask([exInvalidOp, exDenormalized, exZeroDivide,
                  exOverflow, exUnderflow, exPrecision]);
```

before the first GTK call. The `hello_window` example does this before
`g_application_run`.
